# AR Payment Matcher — หา SO จากยอดเงินที่ได้รับ

วันที่: 2026-06-25
สถานะ: design (รออนุมัติก่อนทำแผน)

## ปัญหา

ลูกค้าโอนเงินก้อนเดียวมาจ่ายหลาย SO รวมกัน แต่ไม่ได้บอกว่าเป็น SO ใบไหนบ้าง
ผู้ใช้ต้องนั่งไล่เดาเองว่ายอดที่ได้รับ = SO ใบไหนรวมกัน ทำให้ตรวจสอบยากและช้า

ปุ่ม "รับชำระรวม" เดิมใน [ARAP.jsx](../../../src/components/Finance/ARAP.jsx) แก้เคสนี้ได้แค่บางส่วน —
มันให้เลือกลูกค้าแล้ว "ติ๊กเอง" ว่าจะรับ SO ใบไหน แต่ยังไม่ช่วยตอบคำถามว่า "ยอดก้อนนี้คือ SO ใบไหน"

## ขอบเขต / เงื่อนไขที่ยืนยันแล้ว

- ยอดที่ได้รับ ≈ ผลรวม SO **เต็มจำนวน** เพี้ยนได้แค่เศษ/ค่าธรรมเนียมโอน (ไม่กี่บาท)
  - ไม่มีเคสหักภาษี ณ ที่จ่าย 3%, ไม่มีจ่ายบางส่วนของใบ
- ผู้ใช้รู้ตัวลูกค้าเกือบทุกครั้ง (จากชื่อผู้โอน/สลิป) → ค้นหาจำกัดอยู่แค่ SO ค้างของลูกค้าเจ้านั้น
- แนวทางที่เลือก: **A — เสริมความสามารถ auto-match เข้าไปใน modal "รับชำระรวม" เดิม** (ไม่ทำหน้าแยกใหม่)

## เป้าหมาย (success criteria)

- ใส่ลูกค้า + ยอดที่ได้รับ แล้วระบบบอกได้ว่า "ชุด SO ใดรวมกันได้เท่ายอดนี้ (เผื่อเศษ)"
- กดใช้ชุดที่เจอแล้วรับเงินจบในที่เดียว โดยใช้ logic บันทึกเดิม (`saveBatch`) ทั้งหมด
- เปิดใช้แค่ "เช็คเฉยๆ" (ไม่บันทึก) ก็ได้

## ส่วนที่ไม่อยู่ในขอบเขตรอบนี้ (YAGNI)

- การบันทึก "ยอดรับจริง" ที่ต่างจากยอด SO แล้วปัดเศษที่เหลือ — รอบนี้ถือว่าส่วนต่างเป็นค่าธรรมเนียมธนาคาร ไม่แตะยอดค้างลูกค้า
- การดึงรายการเงินเข้าจากตารางธนาคาร (`bankTxns`) มาจับคู่อัตโนมัติ
- การเดาลูกค้าจากยอดเมื่อไม่รู้ผู้โอน
- เคสหักภาษี ณ ที่จ่าย / จ่ายบางส่วน

## ดีไซน์

### 1. UI ใน modal "รับชำระรวม" (batchPay)

แทรกแถวใหม่ใต้ช่องเลือกลูกค้า ก่อนรายการ SO ค้างชำระ:

ลำดับใน modal:
1. ลูกค้า (เดิม)
2. **[ใหม่]** ยอดเงินที่ได้รับ `[input]`  ±`[tol input, ค่าเริ่ม 50]` บาท  `[ปุ่ม "หา SO ที่ตรงยอด"]`
3. **[ใหม่ — แสดงเมื่อเจอหลายชุด]** กล่องผลการค้นหา: รายการ "ชุดที่เป็นไปได้" แต่ละแถวกดเลือกได้
4. SO ค้างชำระ (เดิม) — SO ในชุดที่เลือกจะถูกติ๊กอัตโนมัติ
5. ยอดรวมที่เลือก (เดิม)
6. รายการชำระ (เดิม) — บรรทัดแรกถูกเติมยอด = ยอดรวมชุดที่ match
7. ปุ่มบันทึก (เดิม → `saveBatch`)

พฤติกรรม:
- ช่อง "ยอดที่ได้รับ" และปุ่มหา ใช้งานได้ต่อเมื่อเลือกลูกค้าแล้ว (รายการ SO โผล่หลังเลือกลูกค้าอยู่แล้ว)
- "เช็คเฉยๆ": เปิด → เลือกลูกค้า → ใส่ยอด → กดหา → ดูผล → ปิดได้โดยไม่บันทึก

### 2. เครื่องยนต์จับคู่ — `findSOCombos`

ฟังก์ชัน pure ระดับ module (วางนอก component — เทสต์ได้ และเลี่ยง TDZ/inline-component ที่เคยเจอ):

```
findSOCombos(soList, targetSatang, tolSatang) -> rankedCombos[]
```

- input: `soList` = SO ค้างของลูกค้า (ใช้ `Math.max(0, so.remaining)` ต่อใบ), `targetSatang`, `tolSatang`
- คิดเป็นสตางค์ (จำนวนเต็ม ×100 ด้วย `Math.round`) เลี่ยง error ทศนิยม
- หาทุกชุดที่ผลรวมอยู่ในช่วง `[target - tol, target + tol]`
- อัลกอริทึม: DFS + ตัดกิ่ง (เรียง remaining มาก→น้อย, ถ้า runningSum > target + tol ตัดกิ่ง)
- cap จำนวนกิ่ง/ผลลัพธ์ กันกรณี SO ค้างเยอะผิดปกติ (เช่นจำกัด combos ที่เก็บ และ/หรือจำกัดจำนวน SO ที่เข้าอัลกอริทึม)
- output: array ของ `{ soNums: string[], sumSatang, diffSatang }` จัดอันดับ:
  1. `|diff|` น้อยสุด (ตรงเป๊ะ diff=0 ก่อน)
  2. จำนวนใบน้อยสุด
  3. ใบล่าสุดก่อน (ตาม date)
  - เก็บแค่ ~5 ชุดบนสุด
- ครอบคลุม subset ขนาด 1 (เงินก้อน = SO ใบเดียว) จนถึงทั้งหมดรวมกัน

### 3. การกด "ใช้ชุดนี้"

- ติ๊ก SO ของชุดที่เลือก → `setBatchSOs(combo.soNums)`
- เติมยอดบรรทัดชำระแรก = ยอดรวมของชุด (`batchTotal` ของชุดนั้น) เพื่อให้ SO ปิดครบทุกใบเวลาบันทึก ผู้ใช้แค่เลือกวิธีชำระ + บัญชี
- ส่วนต่างเล็กจากค่าโอน/เศษ ไม่แตะ (ดู YAGNI)
- กฎตัดสิน: ผลในช่วง tolerance มี **1 ชุด → ติ๊กให้ทันที** (auto-apply); มี **>1 ชุด → โชว์ลิสต์ให้เลือกก่อนติ๊ก**
- หลังจากนั้น flow เดิมรับช่วง (allocate ข้ามใบ + สร้าง payment/bank txn/เช็ค ผ่าน `saveBatch`)

### 4. เคสขอบ

- ยอดว่าง/ไม่ใช่ตัวเลข → no-op
- ยังไม่เลือกลูกค้า → ปุ่มหา disabled
- ไม่เจอชุดในช่วง tolerance → `runMatch` เรียก `findSOCombos` ซ้ำด้วย tolerance กว้างขึ้น (เช่น ×10 ของ tol ปัจจุบัน) แล้วโชว์ชุดที่ใกล้สุด 1 ชุดเป็นใบ้ พร้อมข้อความ "ไม่เจอชุดที่รวมได้พอดี — ชุดที่ใกล้สุดคือ..." ผู้ใช้ติ๊กเองได้ตามเดิม
  - หมายเหตุ: `findSOCombos` คืน "เฉพาะชุดในช่วง tolerance ที่ส่งเข้าไป" เสมอ — กลไกชุดใกล้สุดคือการเรียกซ้ำด้วย tol กว้างขึ้น ไม่ใช่ logic แยกในฟังก์ชัน
- SO ค้างเยอะผิดปกติ → cap ในอัลกอริทึมให้ยังตอบเร็ว

### 5. State ที่เพิ่มใน ARAP

- `recvAmount` (string) — ยอดที่ได้รับ
- `recvTol` (number/string) — tolerance บาท, ค่าเริ่ม 50
- `matchResults` (combo[]) — ผลค้นหาเมื่อมีหลายชุด
- handler `runMatch()` — อ่าน `recvAmount`/`recvTol`, เรียก `findSOCombos`, ถ้าชุดเดียวติ๊กเลย ถ้าหลายชุด set `matchResults`
- handler `applyCombo(combo)` — ติ๊ก SO + เติมยอดบรรทัดชำระ + เคลียร์ `matchResults`
- reset state เหล่านี้ใน `openBatch()`

## เทสต์

เพิ่ม unit test ให้ `findSOCombos` ใน [Finance.test.tsx](../../../src/components/Finance.test.tsx):
- ตรงเป๊ะ (diff=0) — เจอชุดที่รวมพอดี
- มี tolerance — เจอชุดที่ต่างจาก target ภายใน tol
- เจอหลายชุด — จัดอันดับถูก (|diff| → จำนวนใบ → ใบล่าสุด)
- ไม่เจอในช่วง tolerance — คืน array ว่าง
- SO ใบเดียว = ยอด
- SO ทั้งหมดรวมกัน = ยอด

## ความเสี่ยง / หมายเหตุ

- ใช้ logic บันทึกเดิมทั้งหมด ความเสี่ยงพังของเดิมต่ำ
- modal แน่นขึ้น — จัดลำดับให้ "ใส่ยอด+หา" อยู่บนสุดของส่วนเลือก SO เพื่อให้ flow ลื่น
- ถ้าวันหลังอยากต่อยอดเป็นจับคู่เงินเข้าธนาคารอัตโนมัติ ย้าย `findSOCombos` ไปใช้ซ้ำได้ทันที (เป็น pure function)
